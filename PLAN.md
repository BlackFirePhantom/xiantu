# 仙途 (XianTu) — 工程计划与项目规划

> 文档日期：2026-06-14

---

## 一、项目概述

**仙途**是一款轻量级多人在线修仙文字 RPG，基于 Flask + Flask-SocketIO + SQLite 构建。玩家从青云镇出发，历经九大险地，通过斩妖、修炼、炼丹、锻造等玩法，最终渡劫飞升。

**定位**：适合 2 核 1G 低配服务器部署的休闲放置修仙游戏，支持 50+ 人同时在线。

### 2.3 待解决的技术债务

| 优先级 | 问题 | 影响 |
|--------|------|------|
| **P0** | `app.py` 2682 行单文件，所有逻辑耦合 | 维护困难，新人无法上手 |
| **P0** | 密码使用无盐 SHA-256 | 安全风险 |
| **P0** | 无任何自动化测试 | 回归风险极高 |
| **P1** | Session secret_key 每次重启重新生成 | 重启后所有用户 session 失效 |
| **P1** | 数据库迁移靠 `init_db()` 中的列存在性检查 | 历史数据可能损坏 |
| **P1** | CORS 设为 `*` | 安全隐患 |
| **P2** | 无日志系统，print 调试 | 排查线上问题困难 |
| **P2** | README 中技术栈描述与实际不一致（写的是 Eventlet，实际用 threading） | 文档误导 |

## 三、工程计划（技术改进路线）

### Phase 1：基础加固（建议 1-2 周）

**目标**：解决安全和稳定性问题，不改变功能。

#### 1.1 安全修复
- [ ] 密码哈希改用 `bcrypt` 或 `werkzeug.security.generate_password_hash`（带盐）

#### 2.2 重构原则
- **逐步拆分**：每次只提取一个子系统，确保功能不回退
- **game/ 层**：纯业务逻辑，不依赖 Flask/SocketIO，方便测试
- **handlers/ 层**：只做事件接收、参数校验、调用 game 层、返回结果
- **保留所有 socket 事件名不变**，前端零改动

#### 2.3 测试体系建设
- [ ] 引入 `pytest` + `pytest-cov`
- [ ] 为每个 game/ 子模块编写单元测试

## 六、技术决策建议

### 6.1 保持还是换框架？

**建议：保持当前技术栈不变。**

理由：
- 项目定位是轻量级游戏，Flask + SQLite 完全够用
- 迁移到 Django/FastAPI 的收益不大，成本很高
- 前端保持原生 JS（模块化拆分即可），无需引入 Vue/React
- SQLite 在 50 人并发下表现优异，无需迁移到 PostgreSQL

### 6.2 性能瓶颈预判

| 场景 | 当前承载 | 瓶颈点 | 解决方案 |
|------|----------|--------|----------|
| 同时在线 | 50+ | SocketIO 线程模式 | 若超过 100 人，考虑切换 eventlet/gevent |
| 数据库 | 无限 | SQLite 写锁 | WAL 模式已启用；若写冲突频繁，加写队列 |
| 内存 | ~50MB | 字典缓存（online_users） | 定期清理离线用户 |
| 文件大小 | 单文件 DB | game.db 超 100MB | 定期 VACUUM，归档旧数据 |

### 6.3 安全加固优先级

1. **立即修复**：密码加盐哈希（已有用户数据需迁移）
2. **尽快修复**：固定 secret_key、CORS 限制
3. **计划修复**：CSRF 防护、SocketIO 认证、输入校验
4. **可选增强**：登录频率限制、IP 封禁、管理员操作审计
