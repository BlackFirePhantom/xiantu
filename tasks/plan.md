# 后续更新计划：秘境深化与长期玩法

## 目标

将首期“赤焰秘境”从每周的单人点击循环，发展为公平、可结算、可复玩的多人周目标；之后再以突破流派和宗门协作建立长期养成。

## 架构决策

- 秘境周次继续以 ISO 周为边界，所有奖励和排名按周存档。
- 世界首领的血量扣减、贡献写入和击杀奖励必须在同一 SQLite 事务中完成，避免并发下重复结算。
- 每个新玩法都以独立 `game/` 规则模块、`handlers/` Socket 事件和最小 UI 面板实现，保持可测试、可回滚。

## Phase 1：秘境公平性与结算（优先）

### Task 1：原子化首领结算

**目标：** 用单个数据库事务完成首领伤害、贡献和击杀判定。

**验收：**

- [ ] 并发挑战不会将首领血量减成负数。
- [ ] 限定掉落只会被实际完成击杀的一次操作发放。
- [ ] 有自动化测试覆盖首领已死亡、并发边界和重复请求。

**范围：** `models.py`、`handlers/secret_realm.py`、`tests/test_secret_realm.py`。

### Task 2：周结算与参与奖励

**目标：** 周次结束后根据贡献发放参与、排名和首领击杀奖励。

**验收：**

- [ ] 有贡献的玩家可领取一次结算奖励。
- [ ] 榜首与普通参与者奖励不同且可在 UI 中预览。
- [ ] 新周开始时玩家看到新的进度，旧周记录仍可用于结算审计。

**依赖：** Task 1。

### Task 3：探索事件池

**目标：** 将固定探索收益改为至少三类事件：战利品、风险抉择、临时增益。

**验收：**

- [ ] 每次探索展示事件文本和实际收益/代价。
- [ ] 风险事件有明确失败代价，且不能绕过每周次数限制。
- [ ] 事件结果通过单元测试固定随机源验证。

**依赖：** Task 1。

## Checkpoint：秘境可运营

- [ ] 全量测试通过。
- [ ] 浏览器完成探索、挑战、奖励领取和新周状态验证。
- [ ] 排行和掉落在多次刷新后保持一致。

## Phase 2：角色选择与宗门协作

### Task 4：突破心魔与流派被动

**目标：** 每次大境界突破提供互斥的稳定/冒险/资源型选择，形成角色流派。

**验收：**

- [ ] 选择永久保存并影响至少一个现有系统（战斗、修炼或秘境）。
- [ ] 选择说明、收益和代价对玩家可见。
- [ ] 不能重复选择或重置已有被动。

**依赖：** 无；可与 Task 2 并行。

### Task 5：宗门周目标

**目标：** 将秘境贡献汇总为宗门目标，解锁全体加成或宗门商店库存。

**验收：**

- [ ] 宗门总贡献在状态面板中可见。
- [ ] 达成档位后所有成员获得明确效果。
- [ ] 无宗门角色仍可参加个人秘境，不会被阻断。

**依赖：** Task 2。

## Phase 3：玩家经济与叙事留存

### Task 6：悬赏令与委托

**目标：** 支持发布材料收购、首领讨伐和炼制委托，复用现有物品与结算能力。

**验收：**

- [ ] 发布者的赏金会被托管，完成者只能领取一次。
- [ ] 过期和取消操作会正确退还托管物。
- [ ] 交易记录可审计。

**依赖：** Task 1 的事务模式。

### Task 7：修仙志与赛季回顾

**目标：** 记录突破、首领击杀、稀有掉落和宗门里程碑，生成角色生平和周报。

**验收：**

- [ ] 关键事件会被持久化并按时间显示。
- [ ] 周结算页能展示个人与宗门的高光事件。

**依赖：** Task 2、Task 5。

## 风险与缓解

| 风险 | 缓解方式 |
| --- | --- |
| 首领结算竞态 | 先完成 Task 1 的事务化改造，再做奖励与多人目标。 |
| 数值膨胀 | 所有秘境数值放入集中配置，使用测试覆盖低等级和高等级角色。 |
| 新功能过多导致入口拥挤 | 每期只新增一个主入口；后续系统优先挂接现有秘境、宗门或 NPC 面板。 |

## 建议顺序

1. Task 1：原子化首领结算。
2. Task 2：周结算与参与奖励。
3. Task 3：探索事件池。
4. Task 4 与 Task 5。
5. Task 6 与 Task 7。
